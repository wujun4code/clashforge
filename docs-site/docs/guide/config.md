# 首次配置（从“能跑”到“可长期用”）

第一次配置最容易犯的错是“把所有选项一次改完”。  
更稳的方式是：**先做最小可用配置，跑通后再迭代**。

## 配置目标（按优先级）

1. mihomo 能启动且稳定。
2. 至少一个节点可用。
3. 接管策略可控、可回退。

## 推荐步骤

1. 准备一个可靠配置来源（YAML 或订阅）。
2. 在 Setup 导入并激活。
3. 启动内核，先不接管全网。
4. 验证连通性和出口。
5. 再按顺序开启透明代理与 DNS。

## 配置来源怎么选

| 来源 | 适用场景 | 建议 |
| --- | --- | --- |
| 上传 YAML | 你已有稳定配置 | 首次迁移优先 |
| 粘贴 YAML | 临时调试 | 小改动快，但容易出格式错误 |
| 订阅 URL | 长期维护 | 建议主订阅 + 备用订阅 |
| YAML Overrides | 做小范围覆盖 | 只放必要差异，避免“大杂烩” |

## 订阅实践建议（用户视角）

1. 给每个订阅起清晰名字（例如“主线路-日常”“备用-应急”）。
2. 更新频率别过高，优先遵守供应商限制。
3. 至少保留一个备用来源，避免主订阅失效时全盘不可用。

::: warning 敏感信息保护
订阅 URL、Token、诊断报告都可能含敏感信息。发 Issue 前先脱敏。
:::

## 接管策略（推荐初始值）

| 项目 | 推荐初始值 | 理由 |
| --- | --- | --- |
| 透明代理模式 | 先关闭或手动开启 | 避免第一次就影响全网 |
| 防火墙后端 | `auto` | 让系统自动适配 nftables/iptables |
| 启动时自动接管 | 关闭 | 问题更容易定位 |
| LAN 绕过 | 开启 | 防止管理流量被误接管 |

## DNS 策略（别一步到位）

推荐顺序：

1. 先只验证内核与节点。
2. 再启用 DNS 相关能力。
3. 最后再做 dnsmasq 协作或更深接管。

模式理解：

| 模式 | 适合场景 |
| --- | --- |
| `fake-ip` | 分流复杂、规则依赖域名命中 |
| `redir-host` | 行为更直观，排障更容易 |
| dnsmasq `upstream` | 兼容现有 OpenWrt DNS 流程 |
| dnsmasq `replace` | 接管更深，风险更高，需完整验收 |

## 配置完成的验收标准

至少满足：

1. Web UI 稳定可访问。
2. `status` 显示核心状态正常。
3. `check` 返回连通性结果符合预期。
4. 你知道如何用 `stop` 快速回退。

如果还没满足，先不要开启全网接管，先去 [排障](/guide/troubleshooting) 定位根因。
