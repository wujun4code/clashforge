# 导入来源与配置

ClashForge 的配置由多个来源组合而成：订阅、YAML 文件、自建节点、设备分流规则、DNS/透明代理设置、GeoData 和少量高级覆盖。理解这些来源如何协作，后面排障会轻松很多。

## 配置模型

最终运行的 mihomo 配置大致来自这几层：

```text
基础来源
  -> Clash YAML / 机场订阅 / 节点导出

ClashForge 管理控制台
  -> 订阅缓存、节点池、端口、DNS、透明代理、GeoData

设备分流
  -> 按源 IP/CIDR 生成 rule-provider、shadow proxy-group 和 AND 规则

YAML Overrides
  -> 高级用户的最终覆盖，优先级最高

运行时配置
  -> ClashForge 写入 mihomo 的最终 YAML
```

普通用户主要操作“订阅”和“节点”。设备分流在你需要按电脑、手机、团队角色分出口时再配置。YAML Overrides 只建议熟悉 mihomo 的用户使用。

## 首次配置推荐顺序

1. 先导入一个最确定能用的来源。
2. 启动 mihomo 内核，不急着全局接管。
3. 在概览页测试节点可用性和出口 IP。
4. 再配置设备分流。
5. 最后开启透明代理和 DNS 接管。

## 来源一：机场订阅

**这个功能是干嘛的？**
把机场提供的 Clash、SS、Trojan、VLESS、VMess 等订阅解析为 mihomo 节点，并保存更新记录。

**解决什么问题？**
不用把订阅复制到每台设备。路由器统一拉取、缓存和更新订阅，所有设备使用同一份节点池。

**如何使用？**

1. 从机场后台复制 Clash/YAML 格式订阅 URL。
2. 打开 `http://<路由器IP>:7777`。
3. 进入“配置管理”。
4. 在“订阅”里点击添加。
5. 填写名称、URL、可选 User-Agent、更新间隔。
6. 保存后点击更新订阅。
7. 选择该来源启动或合并到最终配置。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 节点集中显示 | 清楚知道当前可用节点数量 |
| 单源或全量更新 | 订阅过期或节点变更时不用逐设备处理 |
| User-Agent 可配置 | 兼容部分对客户端有要求的订阅服务 |
| 多订阅并存 | 主机场、备用机场、临时节点可统一管理 |

注意事项：

| 注意 | 原因 |
| --- | --- |
| 不要公开订阅链接 | URL 通常包含你的认证 token |
| 订阅异常先在浏览器打开测试 | 可快速判断是链接问题还是路由器网络问题 |
| 机场共享 IP 不适合所有业务 | 关键账号建议使用 Worker 或 VPS |

## 来源二：已有 Clash YAML

**这个功能是干嘛的？**
导入你已经在其他 Clash/mihomo 客户端验证过的 YAML 配置。

**解决什么问题？**
迁移成本最低。你可以先把熟悉的配置迁移到 ClashForge，确认路由器环境能跑，再逐步拆分订阅、设备和规则。

**如何使用？**

1. 进入 Setup 或“配置管理”。
2. 选择上传 YAML 文件，或直接粘贴 YAML 内容。
3. 保存为一个容易识别的名称。
4. 预览运行时配置。
5. 启动 mihomo 后在概览页验证。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 保留原有 proxy-groups 和 rules | 迁移风险低 |
| 可查看运行中配置 | 知道 ClashForge 最终写入了什么 |
| 可叠加设备分流 | 逐步从“全局配置”升级到“按设备策略” |

## 来源三：Cloudflare Worker 节点

**这个功能是干嘛的？**
在你的 Cloudflare 账号下部署 VLESS over WebSocket Worker 节点，并导出为 Clash/mihomo 可用节点。

**解决什么问题？**
当你没有 VPS，或想要一个低成本备用出口时，Worker 节点可以作为机场之外的补充。

**如何使用？**

1. 准备 Cloudflare API Token、Account ID 和一个托管在 Cloudflare 的域名。
2. 进入“出口节点”。
3. 点击“Worker 节点”。
4. 填写 Worker 名称和绑定域名。
5. 等待部署完成。
6. 点击导出 Clash 配置，或在订阅发布中选择该节点。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 不需要 VPS | 成本低，适合轻量使用 |
| 可重新部署 | Worker 脚本可由 ClashForge 维护 |
| 可导出配置 | 能用于路由器、桌面客户端或订阅发布 |

局限：

| 局限 | 说明 |
| --- | --- |
| IP 不固定 | 不适合强依赖固定登录来源的业务 |
| 不是独享 IP | 不能等同于自有 VPS |
| 受 Cloudflare 额度限制 | 批量任务需关注请求量 |

## 来源四：VPS/SSH 节点

**这个功能是干嘛的？**
通过 SSH 管理远程 Linux/VPS 服务器，自动部署 GOST、绑定 Cloudflare DNS、申请 TLS 证书、探测可用性，并导出 Clash 配置。

**解决什么问题？**
关键业务需要固定、独享、可长期使用的出口。机场和 Worker 都无法稳定满足这类需求。

**如何使用？**

1. 准备一台海外 VPS。
2. 准备一个托管在 Cloudflare 的域名。
3. 在“出口节点”点击新增 VPS/SSH 节点。
4. 填写服务器地址、SSH 用户、端口、域名和邮箱。
5. 按页面提示授权 ClashForge 的 SSH 公钥。
6. 点击“校验 SSH 连接”。
7. 执行完整部署并等待证书、DNS、GOST、探测完成。
8. 导出 Clash 配置或加入订阅发布。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 固定独享出口 | 适合支付、广告、账号后台、AI API |
| 自动部署 GOST | 减少手工配置服务器步骤 |
| Cloudflare DNS 绑定 | 域名和证书管理更统一 |
| 节点可探测 | 更容易判断服务器还是本地配置问题 |

## 设备分流配置

**这个功能是干嘛的？**
按源 IP 或 CIDR 把设备分组，并限制某些设备组能使用哪些策略组或节点。

**解决什么问题？**
同一个网络里，不同设备的需求不同：工作电脑需要固定出口，电视只需要流媒体节点，访客设备可能应该直连。

**如何使用？**

1. 给关键设备设置 DHCP 静态租约，保证 IP 不变。
2. 进入“设备分流”。
3. 选择配置来源。
4. 创建设备组，例如“运营电脑”“AI 工作站”“娱乐设备”。
5. 从在线设备列表选择 IP，或手动输入 CIDR。
6. 为该设备组配置策略覆盖。
7. 保存后重新生成并启动配置。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 自动生成设备 rule-provider | 不用手写复杂规则 |
| 为设备生成 shadow proxy-group | 不破坏原始订阅结构 |
| 按源 IP 插入 AND 规则 | 同一域名可因设备不同走不同出口 |
| 策略集中在路由器 | 用户设备不必理解代理规则 |

## DNS 与透明代理基础配置

**这个功能是干嘛的？**
控制流量如何进入 mihomo，以及 DNS 由谁解析。

**解决什么问题？**
路由器级代理最常见的问题不是“没节点”，而是 DNS、规则、防火墙、策略路由互相不一致。

**如何使用？**

首次建议：

| 设置 | 建议 |
| --- | --- |
| 透明代理模式 | 先保守，验证后再启用 |
| 防火墙后端 | `auto` 或页面默认值 |
| DNS 接管 | 先不开，节点正常后再开 |
| 绕过 LAN | 保持启用，避免管理页面失联 |
| 开机自动接管 | 跑稳后再开启 |

确认稳定后，可在“高级管理”里调整 tproxy、redir、tun、DNS fake-ip/redir-host、dnsmasq 共存方式等选项。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 设备不用手动设置代理端口 | 连接路由器即可按规则生效 |
| DNS 与规则一致 | 减少国外网站打不开、国内网站变慢 |
| 接管可停止和恢复 | 出问题能快速退回普通网络 |

## GeoData 与规则集

**这个功能是干嘛的？**
管理 GeoIP、GeoSite、rule-provider 文件，并支持域名/IP 规则搜索。

**解决什么问题？**
规则文件缺失或过期会导致分流错误。比如一个应该走代理的域名直连了，或者国内域名被代理了。

**如何使用？**

1. 进入“路由数据”。
2. 查看 `GeoIP.dat`、`GeoSite.dat` 是否存在。
3. 选择下载代理，点击立即更新。
4. 需要长期维护时开启定时更新。
5. 内核运行后在“配置管理 -> 规则集”查看 rule-provider。
6. 输入域名或 IP 搜索命中的规则。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 规则文件状态可见 | 知道缺失、大小和更新时间 |
| 可通过代理下载 | 路由器直连 GitHub 失败时也能更新 |
| 可搜索规则命中 | 排障从猜测变成证据 |

## YAML Overrides

**这个功能是干嘛的？**
给高级用户提供最终 YAML 覆盖能力，通过 deep-merge 写入 mihomo 配置。

**解决什么问题？**
当 Web UI 暂时没有覆盖某个 mihomo 高级字段时，仍可以手动添加。

**如何使用？**

1. 只在理解 mihomo YAML 的情况下使用。
2. 每次只改少量字段。
3. 修改后查看运行中配置。
4. 启动后先验证一台设备。

**实现效果和价值**

| 效果 | 价值 |
| --- | --- |
| 保留高级可扩展性 | 不被 UI 当前字段限制 |
| 优先级最高 | 能覆盖生成配置中的特定字段 |

::: warning
错误的 Overrides 可能导致 mihomo 无法启动或规则异常。普通用户尽量通过 Web UI 配置。
:::
